# Keep 运动打卡应用 - 需求规格文档 (PRD v1.0)

## 1. 项目概述

Keep 是一个运动打卡激励应用，核心玩法是"管理者"督促"打卡者"坚持每日运动打卡，通过积分和奖品兑换机制增加动力。客户端为联网优先的 H5 PWA 应用（不需要离线支持），同一入口根据角色展示不同功能界面。

## 2. 技术栈

- **前端**: Vite + Vue 3 + Vant 4 + Pinia + Vue Router，PWA 配置
- **后端**: Java 25 + Spring Boot，SQLite (WAL 模式) + Redis
- **图片存储**: 七牛云（私有空间 + 签名临时 URL）
- **部署**: Docker Compose，GitHub Actions 自动化，GitHub Packages 镜像托管
- **时区**: 固定 UTC+8 北京时间

## 2.5 品牌视觉与 UI 规范

### 品牌标识

- Logo 文件: 绿底金色圆形徽章，跑步人形 + 金币元素
- Logo 用于: PWA 图标（多尺寸）、首屏加载动画、登录页品牌展示

### 色彩体系（从 Logo 提取）

- **主色 (Primary)**: 深绿色 -- 体现健康/运动活力，用于主要按钮、导航栏、进度条
- **强调色 (Accent)**: 金色 -- 体现奖励/积分，用于积分数字、奖励提示、成就徽章
- **辅助色**: 白色/浅灰背景，深灰文字，红色用于警告/扣分提示
- 具体色值在实施时从 Logo 精确取色确定

### UI 风格

- **整体基调**: 简洁运动风，圆角卡片布局，适度留白
- **一致性**: 全应用统一的色彩、字体、间距、圆角规范，Vant 4 主题定制覆盖默认样式
- **图标风格**: 线性图标为主，与 Logo 的简洁风格一致
- **动效规范**:
  - 打卡成功: 明确的成功反馈动效（如打勾弹跳/撒花）
  - 页面切换: 轻量的路由过渡动画
  - 进度变化: 进度条/数字的平滑过渡
  - 列表加载: 骨架屏占位
  - 原则: 动效服务于反馈和引导，不做纯装饰性动画，时长控制在 200-400ms

### 首屏加载优化

- **目标**: 消灭白屏，用户打开即看到品牌加载画面
- **实现方案**:
  - `index.html` 内联一个极轻量的 CSS 加载动画（Logo + 品牌色旋转/脉冲动效），不依赖任何 JS/CSS 外部资源
  - Vue 应用挂载成功后移除加载动画
  - 关键 CSS 内联，非关键资源异步加载
- **构建优化**:
  - 路由懒加载（按页面 code splitting）
  - Vant 4 按需引入（tree-shaking，仅打包使用到的组件）
  - 图片资源压缩（Logo 等静态资源用 WebP + 降级）
  - gzip/brotli 压缩（Nginx 层配置）

## 3. 用户角色与关系

- **管理者 (Manager)**: 配置规则、维护奖品、核销订单、监督打卡
- **打卡者 (Checker)**: 每日打卡、查看进度、兑换奖品
- **关系**: 一对多（1 个管理者可绑定 N 个打卡者，1 个打卡者只绑定 1 个管理者）
- **限制**: 一个账号只能是管理者或打卡者，不可兼任
- **数据隔离**: 每对管理者-打卡者的数据独立

## 4. 认证与账户系统

### 4.1 注册

- 必填: 用户名（唯一）、密码、图形验证码
- 选填: 邮箱（填写后需通过邮件验证码验证）
- 注册时强提示：未绑定邮箱将无法找回密码

### 4.2 登录

- 用户名 + 密码 + 图形验证码

### 4.3 角色确定

- 直接访问应用: 注册/登录后进入管理者流程
- 通过邀请链接访问: 注册/登录后自动绑定为对应管理者的打卡者
- 已有账号的用户通过邀请链接访问: 如果是管理者账号，提示角色冲突无法绑定

### 4.4 账户管理

- 修改头像、昵称（随时）
- 修改密码（需验证旧密码）
- 密码找回（通过已绑定邮箱发送重置链接，未绑定邮箱则无法找回）

## 5. 核心业务规则

### 5.1 "一天"的定义

- 一个打卡日 = 当天 `RESET_HOUR`:00:00 至次日 `RESET_HOUR-1`:59:59（UTC+8）
- 默认 RESET_HOUR = 5，即 05:00:00 至次日 04:59:59
- RESET_HOUR 由管理者配置

### 5.2 打卡规则

- 每个打卡日只能打卡 1 次
- 打卡必须上传 1 张照片（拍照或相册选取），可附带简短文字（可选）
- 前端在上传前压缩图片：最长边不超过 1920px，JPEG 质量 0.75，目标 < 500KB
- 打卡后当日不可重复打卡，UI 显示"已打卡"状态

### 5.3 连续打卡与周期

```
配置参数:
- STREAK_DAYS: 连续打卡天数（默认 5）
- POINTS_PER_CYCLE: 每周期积分（默认 1）
```

- 连续打卡 STREAK_DAYS 天 = 完成 1 个周期 = 获得 POINTS_PER_CYCLE 积分
- 周期完成后立即开始下一个周期（不重置连续天数计数）
- 例：STREAK_DAYS=5，连续打卡 12 天 = 完成 2 个周期（第5天、第10天各得积分），当前周期进度 2/5

### 5.4 全勤与额外奖励

```
配置参数:
- BONUS_FULL_COUNT: 额外奖励所需全勤次数（默认 3）
- BONUS_POINTS: 额外奖励积分（默认 1）
```

- 完成 1 个打卡周期 = 1 次"全勤"
- 每累积 BONUS_FULL_COUNT 次全勤，额外获得 BONUS_POINTS 积分
- 累积计数不重置，即第 3、6、9... 次全勤各获 1 次额外奖励（BONUS_FULL_COUNT=3 时）

### 5.5 连续打卡中断

- 任何一个打卡日未打卡 = 连续中断
- 中断后重置：当前周期进度归零、全勤累积次数归零
- 已获得的积分不受影响

### 5.6 惩罚机制

```
配置参数:
- PENALTY_DAYS: 惩罚触发天数（默认 10）
- PENALTY_POINTS: 惩罚扣分（默认 1）
```

- 从最后一次打卡的下一个打卡日起计算未打卡天数
- 连续未打卡达到 PENALTY_DAYS 天时扣减 PENALTY_POINTS 积分
- 每再连续 PENALTY_DAYS 天继续扣减（循环触发）
- 积分最低为 0，不会出现负数
- 从未打卡的新用户：从账号绑定日开始计算

### 5.7 积分系统

- **逐笔追踪**: 每次获得积分记录为一笔，包含获取时间、来源、数量、过期时间
- **有效期**: 积分获取后 `POINTS_VALIDITY` 天有效（默认 365 天），管理者可配置
- **过期处理**: 每日定时任务检查并过期超时积分，记录积分变动流水
- **消费顺序**: 兑换时优先消费最快过期的积分（FIFO by expiration）
- **积分来源**: 周期完成奖励、全勤额外奖励
- **积分消耗**: 奖品兑换、惩罚扣减、自然过期
- **积分还原**: 取消兑换时还原积分，如果还原的积分已超过有效期，需二次确认提示

### 5.8 配置变更策略

- 管理者修改任何配置参数后立即生效
- 当前周期进度按新配置重新计算
- **不追溯补发**: 配置降低门槛时，不对历史周期补发积分/奖励
- 已获得的积分和已完成的订单不受影响
- （UX建议）打卡者端应显示"规则已更新"提示

## 6. 管理者功能

### 6.1 初始引导流程

- 注册后强制进入配置引导
- 必须完成所有必填配置项后才能创建邀请链接
- 配置项参见 5.3-5.7 中的所有配置参数

### 6.2 邀请系统

- 生成一次性邀请链接（URL 含唯一 token）
- 链接有效期 24 小时（超时自动失效）
- 被使用后立即失效
- 可随时生成新的邀请链接
- 管理者可查看已绑定的打卡者列表

### 6.3 奖品管理

- **新增奖品**: 名称（必填）、所需积分（必填）、库存数量（必填）、图片（可选，不传用默认兜底图，兜底图url在spring配置文件中配置）
- **编辑奖品**: 可修改名称、图片、库存、所需积分；积分变更只影响后续兑换
- **下架奖品**: 软删除，不再展示给打卡者，已有未核销订单不受影响
- **库存限制**: 减少库存时不能低于已兑换且未取消的订单数
- **随机盲盒**: 作为普通奖品创建，名称可为"随机盲盒"，核销后由管理者口头宣布实际奖品
- **实时库存**: 管理者可查看每个奖品的剩余库存

### 6.4 订单管理

- 查看所有打卡者的兑换订单列表
- 按订单状态筛选
- 扫描打卡者的核销二维码完成核销（调用摄像头扫码）

### 6.5 打卡监督

- 查看每个打卡者的打卡记录（日历视图 + 列表视图）
- 查看打卡者的当前连续打卡天数和周期进度

## 7. 打卡者功能

### 7.1 首页/仪表盘

- **GitHub 风格打卡日历**: 类似贡献热力图，展示历史打卡情况
- **周期进度条**: 视觉化展示当前 STREAK_DAYS 天循环进度（如 3/5）
- **积分提示**: 还差 N 天可获得 M 积分
- **全勤进度**: 当前全勤累积 X/BONUS_FULL_COUNT，再完成 Y 个全勤可获额外奖励
- **打卡按钮**: 未打卡时显示醒目的大按钮；已打卡时缩小按钮，主要显示"今日已打卡"

### 7.2 打卡操作

- 点击打卡按钮 -> 唤起摄像头拍照或选择相册 -> 可选填写文字 -> 提交
- 提交后显示打卡成功反馈
- 当日已打卡不可重复操作

### 7.3 打卡历史

- **日历视图**（默认）: 打卡日期在日历上显示缩略图
- **列表视图**: 按时间降序展示打卡记录明细（照片 + 文字 + 时间）
- 支持两种视图切换

### 7.4 兑换商城

- 展示管理者维护的可用奖品列表
- UI 状态区分：
  - 可兑换: 积分充足且有库存 -> 正常展示 + 兑换按钮
  - 积分不足: 灰显/标记积分差额
  - 无库存: 显示"已售罄"标记
- 兑换确认弹窗，显示将消耗的积分数

### 7.5 订单列表

- 展示已兑换的所有订单
- 按创建时间降序排列
- 订单状态一键过滤（全部 / 待核销 / 已核销 / 已取消）
- **待核销订单**: 显示核销二维码（静态），支持取消兑换
- **已核销订单**: 显示核销时间，不可取消
- **已取消订单**: 展示取消时间
- 取消兑换时如果积分已过期，弹出二次确认："取消兑换后积分将因过期而无法恢复，确定要取消吗？"

### 7.6 积分流水

- 展示所有积分变动记录（获取、消耗、过期、惩罚扣减、取消还原）
- 每条记录包含：变动类型、变动数量、变动时间、备注
- 顶部提示：N 个积分将在 30 天内过期

### 7.7 个人信息

- 显示自己的头像、昵称、用户名
- 显示当前绑定的管理者（头像 + 昵称）
- 编辑头像、昵称入口
- 修改密码入口
- 绑定/修改邮箱入口

## 8. 订单与核销流程

```
兑换 -> [待核销] -> 管理者扫码核销 -> [已核销]
                 \-> 打卡者取消 -> [已取消]（积分还原，库存还原）
```

- 核销二维码: 订单创建时生成，内容为包含订单 ID + 随机验证 token 的 URL
- 管理者扫码后进入核销确认页，确认后完成核销
- 已核销订单不可取消

## 9. 技术要求

### 9.1 图片上传流程

1. 前端压缩图片（max 1920px 长边，JPEG 0.75 质量，< 500KB）
2. 前端请求后端获取七牛云上传凭证（带过期时间）
3. 前端直传七牛云
4. 前端将文件 key 提交给后端保存
5. 后端按需生成签名临时 URL 供前端展示（URL 有效期建议 1 小时）

### 9.2 七牛云文件生命周期管理

核心原则：只有当图片确认不再有任何页面入口可展示时，才删除七牛云上的文件。

需要删除旧文件的场景：

- **用户更换头像**: 个人信息页只展示当前头像，旧头像无任何展示入口，保存新头像后异步删除旧文件
- **管理者更换奖品图片**: 奖品详情和订单详情均展示当前图片，旧图片无展示入口，保存新图片后异步删除旧文件

不删除的场景：

- **奖品下架（软删除）**: 历史订单详情中仍需展示奖品图片，不删除
- **打卡照片**: 第一期不支持删除打卡记录，不涉及清理

实现要点：

- 删除操作异步执行，不阻塞主流程
- 删除失败时记录日志，可通过定时任务兜底重试
- 数据库中记录每个文件的 Qiniu file key，便于追踪

### 9.3 定时任务

- **积分过期检查**: 每日 RESET_HOUR 执行，标记过期积分并记录流水
- **惩罚计算**: 每日 RESET_HOUR 执行，检查未打卡天数并执行扣分

### 9.4 部署架构

```
[Nginx + SSL] -> [keep-frontend 容器] (静态资源)
              -> [keep-server 容器] (API)
              -> [Redis 容器]
              -> [SQLite 文件] (Docker volume)
```

- 前端: Nginx 提供静态文件服务，Docker 镜像基于精简版 nginx:alpine
- 后端: Java 25 精简镜像，SQLite 数据文件通过 Docker volume 持久化
- 前端采用滚动部署（无状态，可多实例）
- 后端采用 health check + 短暂重叠窗口策略（SQLite WAL 模式，busy_timeout 5s）
- GitHub Actions: 构建 -> 推送镜像到 GitHub Packages -> SSH 到目标机器 -> docker compose pull + up

### 9.5 安全

- JWT token 认证，合理过期时间
- 图形验证码防暴力破解
- 七牛云私有空间 + 签名临时 URL 防盗链
- QR 码包含随机 token 防伪造
- 接口限流（登录、注册等敏感操作）
- HTTPS（Let's Encrypt 或自有证书）

### 9.6 单元测试

核心业务逻辑必须覆盖单元测试，确保正确性和可维护性。

**后端（JUnit 5 + Mockito）需覆盖的核心模块：**

- **打卡日边界计算**: RESET_HOUR 参数下的"当前打卡日"判定、跨日边界场景（如 4:59 AM vs 5:00 AM）
- **连续打卡与周期引擎**: 连续天数累积、周期完成判定、中断后重置、配置变更后的进度重算
- **全勤与额外奖励计算**: 全勤次数累积、奖励触发点（第3/6/9次）、中断后重置
- **积分引擎**: 积分发放、FIFO 过期消费、积分过期处理、惩罚扣减（最低为0）、取消兑换后积分还原（含过期判定）
- **惩罚机制**: 连续未打卡天数计算、循环触发扣分、积分不足时的边界处理
- **订单生命周期**: 兑换扣库存/扣积分、取消还原库存/还原积分、核销状态流转、核销后不可取消
- **库存管理**: 扣减/还原、不能低于已占用数量
- **邀请链接**: 生成、过期（24h）、一次性使用失效

**前端（Vitest）需覆盖的核心工具函数：**

- **打卡日边界计算**: 与后端一致的"当前打卡日"判定逻辑
- **图片压缩工具**: 压缩参数验证、输出格式校验
- **积分展示格式化**: 过期提示计算、FIFO 消费预览

**测试原则：**

- 每个阶段的核心逻辑随代码同步编写测试，不单独拆分测试阶段
- 后端 CI 流水线中集成 `mvn test`，测试不通过则阻断构建
- 重点覆盖边界条件和异常路径，不追求覆盖率数字

## 10. 第一期不做（Out of Scope）

- PC 端超级管理员后台
- 离线使用
- 补卡功能
- 多次打卡
- 通知系统（推送/邮件提醒）
- 解绑管理者
- 一人多角色
- 数据导出/备份
- 账号注销
- 国际化
